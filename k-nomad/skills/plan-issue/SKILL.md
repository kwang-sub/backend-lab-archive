# Plan GitHub Issue

GitHub 이슈 번호 또는 이슈 본문을 기준으로 구현 전에 검토 가능한 작업 계획을 작성합니다.

## 작업 순서

### 1. 이슈 조회

- GitHub에서 대상 이슈를 조회합니다.
- 이슈 제목, 본문, 라벨, 체크리스트, 관련 논의가 있으면 함께 확인합니다.
- 이슈 내용이 부족하면 저장소 코드와 현재 구조를 함께 참고해 의도를 추론합니다.

### 2. 문제 이해

- 이슈가 해결하려는 핵심 문제를 1문장으로 요약합니다.
- 버그인지, 기능 추가인지, 리팩터링인지, 문서 작업인지 먼저 분류합니다.
- 영향받는 화면, 모듈, 데이터 흐름, 사용자 시나리오를 식별합니다.
- 불명확한 요구사항, 기술적 리스크, 확인이 필요한 가정을 분리합니다.

### 3. 해결 계획 수립

- 구현 전에 먼저 상세한 실행 계획을 작성합니다.
- 계획은 바로 작업 가능한 수준으로 구체적으로 작성합니다.
- 필요하다면 조사, 구현, 테스트, 검증 단계를 분리합니다.
- 각 단계마다 목적, 예상 변경 범위, 검증 방법을 포함합니다.
- 선행조건과 의존성을 반영해 실행 순서대로 정렬합니다.
- 과도하게 추상적인 표현은 피하고 실제 수정 대상에 가까운 표현을 사용합니다.

### 4. 사용자 검토 요청

- 계획을 작성한 뒤 바로 구현하지 않습니다.
- 먼저 사용자에게 검토를 요청합니다.
- 계획 중 불확실하거나 선택지가 있는 부분은 함께 표시합니다.
- 사용자가 승인하거나 수정 의견을 주기 전까지는 구현 단계로 넘어가지 않습니다.
