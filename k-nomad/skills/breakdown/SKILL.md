# Break Down Problem

사용자가 설명한 문제를 실행 가능한 작은 작업 단위로 세분화합니다.

## 목표

- 먼저 문제를 1문장으로 요약합니다.
- 이 문제가 요구하는 최종 산출물을 명확히 정의합니다.

## 분해 기준

- 문제를 바로 실행 가능한 작업 단위로 나눕니다.
- 각 작업은 너무 크거나 추상적이면 안 됩니다.
- 각 작업은 가능하면 30분에서 2시간 안에 처리할 수 있는 수준으로 작성합니다.
- 기능, 사용자 흐름, 데이터 흐름, 모듈 경계, 위험도와 불확실성을 기준으로 적절히 분해합니다.
- 단순히 순서를 나열하지 말고 왜 분리되는 작업인지 드러나게 작성합니다.

## 각 작업에 포함할 요소

각 하위 작업마다 반드시 아래 항목을 포함합니다.

- 작업명
- 목적
- 선행조건
- 완료 기준
- 검증 방법
- 우선순위

## 의존성과 우선순위

- 어떤 작업이 다른 작업에 의존하는지 명시합니다.
- 선행 작업이 필요한 항목은 의존 관계를 분명히 적습니다.
- 구현 순서대로 정렬합니다.

## 제약 반영

- 문제 설명에 포함된 기술적 제약, 시간 제약, 파일 범위, 품질 요구사항이 있다면 분해에 반영합니다.
- 제약이 명확하지 않다면 일반적인 소프트웨어 작업 기준에서 현실적인 가정을 세우고, 그 가정을 따로 적습니다.

## 출력 제한

- 하위 작업은 최대 7개까지만 제시합니다.
- 너무 작은 단위로 과분해하지 않습니다.
- 추상적인 표현 대신 바로 수행 가능한 문장으로 작성합니다.

## 출력 형식

아래 형식을 그대로 따릅니다.

### 1. 목표 요약
- ...

### 2. 가정 및 제약
- ...

### 3. 작업 분해
1. 작업명: ...
   목적: ...
   선행조건: ...
   완료 기준: ...
   검증 방법: ...
   우선순위: ...

### 4. 의존성 및 권장 순서
- ...

### 5. 바로 시작할 첫 단계
- ...

문제가 너무 광범위하면 먼저 상위 단계로 한 번 분해한 뒤, 가장 먼저 착수할 영역만 추가로 한 단계 더 구체화합니다.
