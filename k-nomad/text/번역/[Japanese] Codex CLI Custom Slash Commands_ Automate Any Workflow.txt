こんにちは。この動画では Codex CLI で非常に強力なのに意外と使われていない機能、カスタム・スラッシュコマンドを紹介します。画面には標準で入っているコマンドが並んでおり、まもなくいくつかを実演しますが、本当の価値は好きなタスクやワークフローに合わせて独自のコマンドをインストールできる点にあります。そうすることで時間を大幅に節約し、強力なワークフローを構築できます。動画を見終えるころには仕組みを理解し、最初のカスタムコマンドをゼロから作成できるようになります。

まず、スラッシュコマンドが何かわからない人や Codex CLI を初めて使う人のために、標準搭載のコマンドを簡単に確認します。命令リストには利用可能なコマンドと各コマンドの説明が書かれています。画面上部には現在利用しているモデル名が表示され、ここでは GPT-5 Codex high です。モデルを変更したいときは `/model` コマンドを使います。スラッシュキーを押すとコマンドパレットが開くので、model を選んで Enter を押せばモデル選択画面が現れ、そのまま推論レベル（低・中・高）も指定できます。今回のデモでは high を選択しました。よく使う `/approvals` コマンドでは承認モードを切り替えられます。レベル 1 は読み取り専用、レベル 2 は自動承認でファイルの読み書きとコマンド実行を許可、フルアクセスは作業中のプロジェクトだけでなくマシン全体のファイルも変更できます。今回はレベル 2 を使用します。

コマンドをインストールできる範囲はグローバルとプロジェクトの 2 種類です。グローバルに入れればマシン上のすべてのプロジェクトから呼び出せます。プロジェクトレベルで入れると、そのプロジェクトの内部でのみ利用可能です。私は複数プロジェクトで再利用するコマンドが多いので、基本的にグローバルにインストールします。ただし企業案件のようにセンシティブな情報を含み、グローバルで公開したくない場合はプロジェクト単位が向いています。この動画ではすべてグローバルコマンドとして追加します。

ではカスタム・スラッシュコマンドのインストール手順を見ていきましょう。方法はいくつかあり、インストール後にバックエンドの構造もお見せします。私がおすすめする最も簡単な方法は Codex とチャットを開いて「○○というカスタムコマンドを作りたい」と伝えることです。すると Codex が必要なファイルを作成し、設定もすべて行ってくれます。手作業で構築することもできますが、Codex に任せた方がずっと速いです。インストールしたコマンドを確認したいときはスラッシュを押して `prompts` と入力すればマシンに登録されたカスタムコマンドの一覧が表示されます。現在は `/code-review` だけが登録されています。

続いて、このコマンドの背後にあるファイルを確認します。`~/.codex/prompts/code-review.md` の内容を Codex に表示してもらうと、このディレクトリがグローバルコマンドの保存場所であり、各コマンドが 1 つの Markdown ファイルとして存在することがわかります。任意のスラッシュコマンドを実行すると Codex は対応する Markdown ファイルを読み込み、そこに書かれた指示とコンテキストに従って処理を進めます。たとえば `/code-review` を呼び出すと `code-review.md` を読み、指定されたファイルやディレクトリを徹底的にレビューします。ファイルには「対象ファイル/ディレクトリの詳細なコードレビューを行う」という目的、重点領域（コード品質・可読性、潜在的なバグやエッジケースなど）、レビュー手順、出力フォーマットが記載されており、Critical・High・Medium・Low の優先度、良い点の報告などが要求されています。コマンド後ろの第 1 引数を `$1` というプレースホルダーで指定し、ターゲットが与えられない場合は「現在のディレクトリにあるすべてのコードファイルをレビューする」というフォールバックも定義されています。

まとめると、カスタムスラッシュコマンドは、繰り返し行う作業や長文プロンプトを自動化するための仕組みです。複雑なプロンプトを Markdown に書いてコマンドとして登録しておけば、毎回長い指示を入力したりコピー＆ペーストしたりせずに済みます。では `/code-review` がどう動くか試してみましょう。デモ用に複数の問題を仕込んだ `messy-code.js` を用意しました。`/code-review messy-code.js` を実行すると、先ほどの Markdown をコンテキストとして読み込み、ファイルを分析します。結果には Critical/High/Medium/Low の順に問題が並び、コードの良い点も記載されます。

そのまま「見つけた問題をすべて修正して」と伝えると、Codex は計画を立て `messy-code.js` を修正します。完了後は何を直したかをまとめ、必要に応じて追加の提案もしてくれます。AI が生成したコード（いわゆる vibe-coded なプロジェクト）を定期的に検査し、健全性を保ちたいときに非常に役立つワークフローです。

私はよく使うコマンドをパックにまとめて公開しています。たとえば snapperai.io/codex-prompt-packs では Codex CLI 用のコマンドパックを配布しており、ダウンロードするとコピペだけで複数のコマンドをインストールできるプロンプトが付属しています。この動画で紹介するのは「Code Analyzer Pack」です。`/code-review` のほかに `/security-audit`、`/refactor-finder`、`/dependency-check`、`/generate-tests` の 5 つが含まれます。Security Audit は SQL インジェクションなどの脆弱性をスキャンし、Refactor Finder はリファクタリング候補を見つけ、Dependency Check は古いパッケージを知らせ、Generate Tests は指定したファイルやディレクトリ向けのテストを生成します。これらのプロンプトは実案件で検証済みなので、どんな指示を書けばいいのか悩む必要がありません。フォームに氏名とメールアドレスを入力すれば、ダウンロードリンクがメールで届きます。

インストールファイルを開くと、このパックにはコード解析向けの強力なコマンド 5 種類が含まれ、あらゆる言語で動作し、30 秒でセットアップでき、プロダクションで鍛えられていると説明されています。Step 1 では「下記のマスターインストールプロンプトをコピーして Codex CLI に貼り付ける」とあります。`markdown` という語からコピーを始めると、グローバルの prompts ディレクトリに 5 つのカスタムコマンドを作成するよう指示する文章が続きます。各コマンドは `code-review.md` などの Markdown ファイルとして保存され、ファイル名がそのままコマンド名になります。本文には各コマンドが達成すべき目的、出力形式、遵守すべき手順が詳細に書かれています。たとえば `security-audit.md` には「指定したコードに対し包括的なセキュリティ監査を行い、SQL インジェクションや環境変数の漏洩、弱いハッシュ化などを検出し、重大度・場所・脆弱性タイプ・リスク説明・推奨修正案を報告すること」と明記されています。どのコマンドも `$1` を使って最初の引数から対象ファイル/ディレクトリを受け取り、必要に応じてフォールバック動作も定義されています。プロンプトは「5 つすべてのコマンドが正しく作成されたことを確認せよ」という文で終わり、そこまでをコピーすれば準備完了です。

Step 2 は Codex を再起動すること。新しいコマンドは再起動しないと一覧に表示されません。実際に、コピーしたプロンプトを Codex チャットに貼り付けて実行すると、約 55 秒で prompts フォルダに 5 つの Markdown が生成され、検証まで完了したとのメッセージが返ってきます。その後セッションを再起動して `/prompts` を入力すると、`code-review`、`dependency-check`、`generate-tests`、`refactor-finder`、`security-audit` が一覧に表示されます。

次に `/refactor-finder` を試しましょう。このコマンドはカレントディレクトリのコードベース全体を解析し、リファクタリングの機会を提示します。デモプロジェクトで実行したところ 58 秒ほどで結果が返り、「合計 8 件の機会を特定、推定テクニカルデット 6.5 人日、すぐに着手できる Quick Win が 3 件」というサマリーとともに、検証フローの平坦化、割引階層の統合、HTTP クライアントの利用統一などの提案が示されました。Critical Issue には SQL インジェクション、危険な DB アクセス、ログへの秘密情報混入、弱いパスワードハッシュ、決済ログにカード情報が残る問題などが挙げられ、高優先度の問題とクイックウィンも区別されています。レスポンス末尾には「サンプルのリファクタリング案や作業順序が必要なら知らせてほしい」と書かれています。特に AI が書いたコードが含まれて品質が心配なプロジェクトでは、この種のコマンドが非常に役立ちます。

最後に `/security-audit` を実行します。新機能やファイルを追加するたびに、パスワードや API キーの漏洩、認証まわりの脆弱性がないかを確認するために定期的に走らせるべきコマンドです。例として `auth-broken.js` に対して実行すると、わずか 14 秒で複数の問題を報告しました。出力には重大度、該当行番号、脆弱性タイプ（SQL インジェクションなど）、リスク説明、推奨修正案と修正コードが含まれています。最後に「モジュールを安全に書き直したり、自動のシークレットスキャンテストを追加したりする手伝いが必要なら教えてください」と書かれているので、「モジュールを安全に書き直して」と返すと、Codex が計画を立ててファイルを修正します。

修正後に同じファイルで再度 `/security-audit` を走らせると、今回は「再チェックの結果、悪用可能な脆弱性は検出されなかった」と報告され、現在のセキュリティ状況や実装済みの予防策、さらなる多層防御を希望するかどうかが示されます。このように、1 つの Codex CLI セッション内で問題検出・修正・再検証まで完結できるのがカスタムスラッシュコマンドの強みです。毎回長いプロンプトを書く必要はなく、保存済みのコマンドを呼び出すだけで幅広いファイルに対して素早く監査を行えます。

まとめとして、紹介した Code Analyzer Pack を試したい場合は snapperai.io/codex-prompt-packs にアクセスし、名前とメールアドレスを入力してダウンロードリンクを受け取ってください。そこに含まれる Markdown を Codex CLI に貼り付ければ、5 つのコマンドを一瞬でインストールできます。もちろん Codex CLI と直接会話しながら自分だけのプロンプトを定義してインストールすることも可能です。次回は PRD Wizard Pack を使って PRD を自動生成する方法を紹介する予定です。私のサイトで人気だった PRD テンプレートをベースにした強力なワークフローなので、興味があればぜひチャンネル登録してお待ちください。ご視聴ありがとうございました。
